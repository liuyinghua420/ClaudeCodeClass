## Problem Statement

Teaching zero-background adults to use Claude Code productively in 8 in-person sessions (2 hours each). Students are a mixed group: some professionals wanting AI in their jobs, some aspiring builders, some curious. By session 8, every student should have built a real project using Claude Code.

## What Makes This Cool

This isn't another "learn to code" course. It's a "learn to direct an AI agent" course. The mental model shift is: you're not learning a programming language, you're learning to manage a very fast junior developer who reads your entire codebase, makes decisions, and executes. That reframing makes Claude Code accessible to non-technical people in a way that traditional coding courses never could.

The gap in the market: every vibe coding course teaches prompt → code. Nobody is teaching the agent management mental model (context management, iterative refinement, evaluating AI output, knowing when to accept vs. push back). That's the real skill.

## Constraints

- 8 sessions x 2 hours each, in-person with laptops
- **Class size:** 10-15 students, 1 teacher + 2 helpers (1 helper per 5-7 students)
- Students: zero background, mixed motivations, Chinese-speaking adults
- **Language:** Instruction in Chinese. Tool interfaces (terminal, Claude Code) remain in English. Handouts and slides bilingual (Chinese explanations, English commands/code).
- **Break schedule:** 10-minute break at the ~55-minute mark of each session. All session timings account for this.
- Must produce a tangible project by session 8
- Environment setup must be bulletproofed (this is where beginner courses die)
- Teacher delivers personally, not recorded/async

### Technology Scope

All projects are client-side web applications (HTML/CSS/JavaScript) opened in a browser. No server, no database, no backend required. This keeps the feedback loop instant: save file → refresh browser → see result.

### Student Prerequisites

- Laptop with macOS 10.15+ or Windows 10+ (no tablets, no Chromebooks)
- Admin/install rights on the machine (can install software without IT approval)
- Stable internet connection (school/venue Wi-Fi or personal hotspot)
- **No coding experience required**

### API Key Strategy

Teacher provides shared API keys for all in-class use. Students do NOT need to create their own Anthropic accounts or add payment methods during the course. This removes the biggest friction point for beginners.

- **Default model:** Claude Sonnet (fast, affordable, good enough for all student exercises)
- **Teacher demos:** Claude Opus (higher quality for the "wow moment" and live demos)
- **Cost estimate:** ~$150-250 total for 8 sessions with 10-15 students at Sonnet pricing. Budget $300 with buffer.
- **Concurrency:** Provision 2-3 API keys across separate accounts and distribute across student groups (Key A for students 1-5, Key B for 6-10, Key C for 11-15). This avoids per-key rate limits when 15 students send requests simultaneously. Test concurrency ceiling before Session 1.
- **Post-course:** Students who want to continue get a guide for setting up their own accounts.

### Helper Briefing

Before the first session, brief helpers (30 minutes):
- Walk through the setup guide step by step
- Review common failure modes: Node.js version conflicts, PATH issues, permissions errors on Windows
- Core principle: unblock students, don't build for them. Help them type the command, don't type it yourself.
- Assign zones: each helper covers 5-7 students in a specific area of the room

## Premises

1. **Project-first, not theory-first.** Mental models emerge from doing. Theory sessions without immediate hands-on lose mixed groups fast.
2. **Environment setup deserves its own time.** Getting Node.js + Claude Code + terminal working is a full lesson for beginners.
3. **Students learn to direct AI, not to code from scratch.** The skill is prompting, iterating, debugging-with-AI, and evaluating output.
4. **8 sessions give breathing room.** Zero background + project outcome + mixed group needs space for catch-up and individual pacing.

## Approaches Considered

### Approach A: Linear Build
One project built progressively across sessions 4-8. Simple progression but inflexible for a mixed group. If the project doesn't resonate, students are stuck.

### Approach B: Milestone Sprints (CHOSEN)
Guided → semi-guided → independent. Session 1 delivers a "wow moment." Sessions 3-4 build a guided mini-project. Sessions 5-6 offer choice. Sessions 7-8 are personal projects with coaching. Builds real autonomy progressively.

### Approach C: Case-Study Driven
Each session is a standalone case study. Portfolio of 4-5 small projects. Most prep work, but every session is self-contained. Risk: students feel like they're doing exercises, not building "their thing."

## Recommended Approach: Milestone Sprints

### Scaffolding Levels

- **Guided** (Sessions 3-4): Teacher-led step-by-step walkthrough. Checkpoints every 10-15 minutes ("by now you should see X"). Teacher demos each step before students try it. Students build the same project type with shared checkpoints, but personalize features within that type.
- **Semi-guided** (Sessions 5-6): Students work from a project template independently. Template includes: a project brief, a suggested build order, and 3-4 milestone checkpoints. Teacher circulates and helps on request, not proactively.
- **Independent** (Sessions 7-8): Students conceive, scope, and build their own project. Teacher coaches on request. No step-by-step guide.

**Session timing:** Each "Hour" below is ~55 minutes of instruction. A 10-minute break falls between them. Total instruction time per session: 110 minutes.

### Session 1: "Wow Moment" + Environment Setup (2 hrs)

**Hour 1: The Wow Moment (do this FIRST, before any setup)**
- Teacher demos Claude Code live: takes a student's idea from the audience and builds a working web page in 10 minutes
- **Idea steering:** Ask for ideas, then select/reshape one that fits: a single-page, static, visual result (e.g., a birthday card, a personal homepage, a restaurant menu). Reject anything requiring a backend or login.
- Students see the full loop: describe what you want → watch AI build it → see it in a browser
- This is the "I want to do that" moment. Everything else is scaffolding to get there.
- **Fallback:** Teacher has 2-3 pre-tested demo ideas ready. If the API fails, show a pre-recorded screencast of the same flow.

**Hour 2: Environment Setup (Start)**
- Pre-prepared setup guide (printed bilingual handout + screen)
- Install checklist: Terminal basics → Node.js → Claude Code → configure shared API key
- **Critical:** 2 helpers walk the room. Pre-configured USB drives as primary backup.
- **USB contents:** Node.js LTS installer (macOS + Windows), Claude Code npm package tarball for offline install, setup guide PDF, cheat sheet PDF.
- Goal: get as many students through the install as possible
- Students who finish early: try 3 commands from the provided cheat sheet

**Fallback plan:** Students who can't install locally use claude.ai/code (web version) immediately. Nobody leaves session 1 without having interacted with Claude. Setup stragglers finish in the first 20 minutes of Session 2 (built-in buffer).

### Session 2: Setup Buffer + AI Mental Models + First Hands-On (2 hrs)

**First 20 minutes: Setup Completion Buffer**
- Students who didn't finish setup in Session 1 complete it now (helpers assist)
- Students who are set up: free exploration time with Claude Code, try commands from cheat sheet

**Next 40 minutes: How to Think About AI**
- Mental model 1: Claude Code is a junior developer. You're the manager. You give direction, it executes, you review.
- Mental model 2: Context is everything. The AI can only work with what it can see. Show how project structure, file names, and clear instructions change output quality.
- Mental model 3: Iteration, not perfection. First output is a draft. The skill is knowing how to push it toward what you want.
- Demo: same prompt, bad context vs. good context. Show the dramatic difference.

**Hour 2: Directed Practice**
- Students practice the core loop: describe → generate → evaluate → refine
- Exercise 1: Build a personal introduction page (HTML) using Claude Code
- Exercise 2: Modify it. Change colors, add a section, fix something that looks wrong
- Key skill: reading Claude's output and deciding "yes, that's right" or "no, change this"

### Session 3: Mini-Project 1 — Guided Build, Part 1 (2 hrs)

**Project: A personal daily planner (everyone builds the same tool type)**
- Everyone builds a daily planner, but personalized: some add meal tracking, some add exercise, some add work tasks
- Same tool type = same checkpoints work for everyone. Personalization keeps it interesting.
- Session 3 focuses on: project structure, describing requirements to Claude Code, building the core functionality

**Hour 1: Requirements Thinking**
- How to break a big idea into small, buildable pieces
- Exercise: write a 5-bullet description of what the tool should do (in plain language)
- Feed it to Claude Code. Watch what happens. Evaluate.

**Hour 2: Building the Core**
- Students work through the guided build, step by step
- Teacher walks the room, helps with stuck points
- Key teaching moment: when Claude Code does something unexpected, how to course-correct

### Session 4: Mini-Project 1 — Guided Build, Part 2 (2 hrs)

**Hour 1: Adding Features**
- Continue building: add 2-3 features to the tool
- Teach: how to give Claude Code context about existing code ("read this file first, then add...")
- Practice: multi-turn interactions, building on previous work

**Hour 2: Debugging + Polish**
- Introduce debugging with AI: "this doesn't work, here's what I expected, here's what happened"
- Students fix issues in their projects
- Polish: make it look good, add finishing touches
- **Milestone: everyone has a working tool they built themselves**

### Session 5: Mini-Project 2 — Semi-Guided, Part 1 (2 hrs)

**Students choose from 3-4 project templates:**
- **Option A: Small business landing page** — A one-page website with hero section, services list, contact form, and responsive design. Finished product: a professional-looking page they can actually use or give to someone. Starting scaffold: a project folder with a brief and example screenshots.
- **Option B: Personal portfolio / hobby showcase** — A multi-page site showcasing the student's interests, work, or hobbies. Finished product: something they'd share on WeChat. Starting scaffold: a template with placeholder sections and sample content.
- **Option C: Simple interactive game** — A quiz game, memory card game, or number guessing game. Finished product: something fun they can show friends. Starting scaffold: a basic HTML page with a game loop structure described in the brief.
- **Option D: Utility tool** — A unit converter, countdown timer, todo list, or simple calculator with a twist. Finished product: a single-page web app that does one thing well. Starting scaffold: a project brief with clear input/output spec.

**Hour 1: Planning Your Project**
- Students pick their template, customize requirements
- Practice: writing good project briefs for Claude Code
- Teacher reviews each student's brief, suggests improvements

**Hour 2: Independent Building with Scaffolding**
- Students build using their template as a starting point
- Teacher provides checkpoints: "by minute 30, you should have X working"
- Pair struggling students with faster ones

### Session 6: Mini-Project 2 — Semi-Guided, Part 2 (2 hrs)

**Hour 1: Advanced Patterns**
- Teach: Plan Mode in Claude Code — before writing code, ask Claude to plan its approach first. Use `/plan` or describe what you want and say "plan this before building." This prevents Claude from rushing into the wrong direction on larger projects.
- Teach: How to handle larger projects (file organization, context management)
- Students apply these patterns to their project

**Hour 2: Complete + Share**
- Finish mini-project 2
- Quick showcase: each student shows their project to a neighbor (2-3 min each)
- **Assign homework:** "Come to Session 7 with one idea for a personal project." Hand out the project idea menu now so students can browse it during the week.
- **Milestone: students have built something they chose, not just followed along**

### Session 7: Personal Project — Your Idea (2 hrs)

**Hour 1: Idea Workshop**
- Each student writes down: "One thing in my life/work that could be easier with software"
- **Project idea menu provided:** A list of 10 pre-scoped project ideas students can adopt or adapt. Examples: birthday reminder page, restaurant menu site, workout tracker, meeting notes organizer, class schedule viewer. Students who have their own idea can use it; the menu removes the blank-page problem.
- **Homework from Session 6:** Students should come to Session 7 with a project idea already in mind (assigned at end of Session 6). Session 7 Hour 1 refines, not conceives.
- Teacher helps refine ideas into buildable scope (this is the hardest teaching moment)
- Key rule: if you can't describe it in 5 bullets, shrink it. Teacher veto on scope.
- Students write their project brief

**Hour 2: Build Sprint**
- Students build their personal project
- Teacher coaches individually, moving between desks
- Focus: unblocking, not hand-holding. Let students struggle productively.

### Session 8: Finish + Show & Tell (2 hrs)

**Hour 1: Finish + Polish**
- Final build time for personal projects
- Help students prepare a 2-minute demo (what it does, how they built it, one thing they learned)
- **Deployment is NOT required.** Students demo from their local machine. A post-course guide covers deployment (GitHub Pages, Vercel) for those who want to share their projects online later.

**Hour 2: Demo Day**
- Every student presents their project (2 min each)
- Celebrate what people built
- Teacher provides "where to go from here" resources
- Optional: create a class WeChat/group for continued learning

## Success Criteria

- 95%+ of students can execute a Claude Code command by end of session 1 (local or cloud fallback). Remaining students complete local setup by Session 2.
- 90%+ of students have a working mini-project 1 by end of session 4
- 80%+ of students have a demoable personal project in session 8 (complete or meaningfully in progress)
- Students report feeling "I can do this on my own" in post-course feedback
- At least 3 students continue using Claude Code independently after the course

## Distribution Plan

Course materials (slides, handouts, project templates, setup guides) stored in the ClaudeClass repo. Teacher delivers in-person. Materials not publicly distributed initially, but structured well enough to share or license later if the course proves successful.

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Setup fails for some students | Cloud fallback (claude.ai/code web), pre-configured USB drives, 2 helpers, Session 2 has 20-min setup buffer |
| API costs concern students | Teacher provides shared API keys (~$150-250 total, budget $300). 2-3 keys distributed across student groups to avoid rate limits. |
| Wide skill gap in mixed group | Pair programming (fast + slow), differentiated checkpoints, multiple project difficulty levels |
| Personal project scope creep (session 7) | Hard rule: 5 bullets max. Project idea menu provided. Teacher veto on scope. |
| Students miss a session | Each session pair (3-4, 5-6, 7-8) has a catch-up checkpoint at the start of the second session. Teacher sends session recap + key files via WeChat after each class. Students who miss Part 1 of a pair need to complete the catch-up checkpoint before Part 2. |
| API outage or rate limit | Pre-cached demo outputs for teacher to show if API is down. Offline exercises (code reading, project planning) as backup activities. Backup API key on a separate account with independent rate limits. |
| Student misses Session 1 or 2 | Send setup guide + pre-recorded demo video via WeChat. Session 2 has 20-min setup buffer. Students who miss Session 2 receive mental model handout and complete Exercise 1 at home before Session 3. |

# 第 2 次课：教案规格书

## 课程信息

| 项目 | 内容 |
|------|------|
| 课次 | 第 2 次课（共 8 次） |
| 主题 | 安装缓冲 + AI 思维模型 + 初次动手 |
| 时长 | 2 小时（110 分钟教学 + 10 分钟休息） |
| 目标 | 建立"指挥 AI"的思维方式，掌握核心循环：写任务书 → 生成 → 评估 → 优化 |

## 学员在本课结束时应该达到的状态

1. **全员安装就绪**：100% 学员完成本地 Claude Code 安装（或确认使用网页版备用方案）
2. **掌握四个思维模型**：理解"AI 是初级开发者"、"上下文决定一切"、"先写任务书再动手"、"迭代而非完美"

4. **建立评估能力**：能判断 Claude 的输出"对不对"、"好不好"，并用自然语言提出修改要求

>
> 会对话的人和不会对话的人，用同一个 AI，做出来的东西天差地别。今天之后，你就是那个"会对话的人"。

**要点：**
- 控制在 5 分钟内

- 给学员制造期待感

### 第三部分：AI 思维模型（0:20 - 0:55）

**过渡语：**

> 好，工具都就绪了。接下来 35 分钟，我要教你们这门课最值钱的东西 —— 不是某个命令，而是四种思维方式。
>
> 掌握了这些思维方式，你用 AI 的效果会比别人好 10 倍。这不是夸张。

#### 思维模型 1：Claude Code 是初级开发者，你是经理（10 分钟）

**老师说：**

> 想象你刚招了一个新员工。这个新员工有三个特点：
>
> **第一，速度极快。** 你让他做什么，几秒钟就做完了。
>
> **第二，听话。** 你说什么他就做什么。
>
> **第三，他需要你给方向。** 如果你说"做个好看的东西"，他会做 —— 但做出来可能不是你想要的。如果你说"做个蓝色背景、白色标题、三个段落的页面"，他做出来就差不多了。
>
> Claude Code 就是这样一个"初级开发者"。而你，是它的经理。
>
> 经理做什么？三件事：
> 1. **给方向** —— 告诉它做什么
> 2. **审查** —— 看看做得对不对
> 3. **反馈** —— 告诉它哪里要改
>
> 你不需要知道代码怎么写。你只需要知道"我想要什么"和"这个结果对不对"。

**类比互动：**

> 举个例子。假设你让新员工做一份PPT。你会怎么说？
>
> A："帮我做个PPT。" —— 他做出来大概率不是你想要的。
>
> B："帮我做个 10 页的 PPT，主题是年度总结，第一页放标题和日期，第二页放今年的三个亮点，第三页放数据图表..." —— 好多了，对吧？
>
> 跟 Claude Code 对话也是一样的道理。你说得越清楚，它做得越对。

**关键点：**
- 用"经理 - 初级开发者"类比，学员容易理解
- 重点是降低心理门槛：你不需要会写代码，你只需要会"管理"
- 不要在这里展开太多，点到即止

#### 思维模型 2：上下文决定一切（10 分钟）

**老师说：**

> 第二个思维模型：**上下文决定一切。**
>
> Claude 只知道你告诉它的事情。它看不到你脑子里的画面。
>
> 让我演示一下区别有多大。

**现场对比演示（这是本环节的重头戏）：**

打开终端，创建两个文件夹，分别演示：

**差的上下文：**

```bash
mkdir demo-bad && cd demo-bad
claude
```

```
做一个网页
```

等 Claude 生成后，打开浏览器展示结果。

**老师说：**

> 看看这个结果。能用吗？能。但它完全不知道你想要什么，所以做了一个非常普通的东西。

**好的上下文：**

```bash
cd .. && mkdir demo-good && cd demo-good
claude
```

```
帮我做一个个人介绍网页。我叫小明，是一个摄影爱好者。
要求：
- 页面顶部有一个大标题"小明的世界"
- 标题下面有一段自我介绍，大概 3-4 句话
- 下面分三个板块：我的作品、我的爱好、联系方式
- 风格要文艺一点，用暖色调，字体柔和
- 加一些好看的 emoji 装饰
```

等 Claude 生成后，打开浏览器展示结果。

**老师说：**

> 看到区别了吗？同一个 AI，结果天差地别。唯一的区别就是 —— 你给了它多少信息。
>
> 记住这条规则：**你给 Claude 的信息越多、越具体，它给你的结果越好。**
>
> 这不是说你要写一篇文章。而是你要想清楚：
> - 我想要什么？（功能）
> - 我想要什么样的？（风格）
> - 有没有具体要求？（细节）

**要点：**
- 对比演示一定要提前测试，确保差异足够明显
- 让学员看到**同一个工具**，仅因输入不同就产出截然不同的结果
- 这是本课最重要的"啊哈时刻"

#### 思维模型 3：先写任务书，再让 AI 动手（8 分钟）

**老师说：**

> 第三个思维模型：**先写任务书，再让 AI 动手。**
>
> 刚才我们看到，上下文越清楚结果越好。那问题来了 —— 怎么把上下文"整理清楚"？
>
> 答案很简单：**动手让 AI 做之前，先花 2 分钟把你想要的东西写下来。**
>
> 我给它起个名字，叫"任务书"。就像你给新员工派活，先把需求写在纸上，而不是嘴上一句"你看着办"。

**任务书三件套（投影展示）：**

```
1. 我想要什么？        —— 功能、内容
2. 我想要什么样的？     —— 风格、感觉
3. 有没有具体要求？     —— 细节、限制
```

**老师说：**

> 这三个问题，就是你的任务书。你不需要写得多长，每项一两句话就够。
>
> 为什么要先写？三个原因：
> 1. **逼你自己想清楚。** 很多时候你以为知道自己想要什么，写下来才发现"其实我没想好"。
> 2. **AI 一次就能做对大半。** 省掉反复返工的时间。
> 3. **任务书可以复用。** 下次想做类似的东西，改改任务书就行。

**现场对比（用刚才的小明例子回溯）：**

> 其实我刚才给 Claude 的那段话，就是一份任务书 ——
>
> - 我想要什么？→ 个人介绍页面，主人公是摄影爱好者小明
> - 我想要什么样的？→ 文艺风格、暖色调、字体柔和
> - 有没有具体要求？→ 大标题"小明的世界"、三个板块、加 emoji
>
> 你看，其实不复杂。三个问题回答清楚，任务书就成了。

**关键点：**
- 强调"先写再做"是一个动作，不是一个理论
- 告诉学员：等下练习时，**动手敲命令前先把任务书写在纸上或记事本里**
- 这是业界一个正式做法，叫 spec-driven（按说明书做）—— 不用讲术语，但可以提一句"这是程序员现在用 AI 的标准做法，你们一上来就学的是高级打法"

#### 思维模型 4：迭代，而非追求完美（7 分钟）

**老师说：**

> 第三个思维模型：**迭代，而非完美。**
>
> 很多人第一次用 AI 会有一个误区：觉得要一次把需求说完美，AI 才能做出好东西。
>
> 不是的。事实上，最好的用法是：
> 1. 先说个大概 → 看结果
> 2. 不满意的地方 → 告诉它改
> 3. 再看 → 再改
> 4. 几轮之后 → 就是你想要的了
>
> 这就像改作文。没有人第一稿就是完美的。

**现场演示迭代过程：**

用刚才"好的上下文"生成的页面，继续在 Claude Code 中修改：

```
这个页面不错，但我想改几个地方：
1. 标题太小了，再大一倍
2. 背景色太亮了，换成米白色
3. "联系方式"板块加一个微信二维码的占位图
```

等结果出来，刷新浏览器展示。

```
很好！再改一下：
- 三个板块用卡片式布局，加圆角和阴影
- 页面底部加一行小字："用 AI 制作"
```

再次刷新展示。

**老师说：**

> 看到了吗？我从来没有一次说完所有需求。我是一步步改过来的。
>
> 每一次修改，我只需要说"哪里不对"和"我想要什么样"。
>
> 这就是核心技能：**不是一次说对，而是一步步调对。**
>
> 记住三句话：
> 1. 第一次结果是草稿，不是成品
> 2. 改比重做快 —— 说"把标题改大"比重新描述整个页面快得多
> 3. 每次只改 1-2 个地方，不要一口气提 10 个修改

**要点：**
- 降低学员压力：不需要一次说完美
- 演示迭代时要刻意制造"越来越好"的感觉
- 让学员看到迭代的节奏：改一点 → 看一看 → 再改一点

### 第四部分：引导式练习（1:05 - 1:50）

**过渡语：**

> 下半场是你们的时间。我们要用刚学的三个思维模型，做一个真正能帮你学英语的小项目。

#### 练习 1：基于电子书 txt 文件，做一个 Naval 金句英语学习页面（25 分钟）

**老师说：**

> 每个人做一个自己的英语学习页面。老师已经提供了一份电子书 txt 文件，比如 `naval.txt`。
>
> 你的任务是：让 Claude Code 从这份 txt 里挑出 **Naval 书里 5 句值得学习的话**，并把它们整理成一个适合学习英语的页面。
>
> 每句话至少要有这四样东西：
> 1. 英文原句
> 2. AI 生成的音标
> 3. 中文翻译
> 4. 中文解读（这句话在说什么，为什么值得学）
>
> 但是！动手敲命令之前，先做一件事 —— **写任务书**。
>
> 拿出一张纸，或者打开电脑的记事本，花 3 分钟回答三个问题：
>
> 1. 我想要什么？（从 `naval.txt` 里选 5 句话，做成英语学习页面）
> 2. 我想要什么样的？（简约？卡片式？像背单词 App？什么颜色？）
> 3. 有没有具体要求？（每句是否加关键词高亮？是否标注来源？是否要重点词解释？）
>
> 写完再动手。这 3 分钟花得值。

**投影上显示步骤引导：**

```
第一步：写任务书（纸上或记事本）
  - 我想要什么？
  - 我想要什么样的？
  - 有没有具体要求？

第二步：打开终端
第三步：创建项目文件夹
  mkdir naval-quotes && cd naval-quotes
第四步：准备电子书 txt 文件
  确保老师提供的 naval.txt 在当前文件夹里
第五步：启动 Claude Code
  claude
第六步：把任务书的内容告诉 Claude
  （例如：请基于 naval.txt 选 5 句话，展示原句、音标、中文翻译和解读）
第七步：打开浏览器查看结果
  open index.html    （macOS）
  start index.html   （Windows）
```

**老师的角色：**
- 前 3 分钟：在教室走动，看学员的任务书，不看电脑
- 任务书写得太模糊的学员 → 引导加细节："你说做英语学习页面，那 5 句话要偏短一点还是偏有启发性？"
- 任务书写得过于复杂的学员 → 引导简化："先做一个简单版本，后面再加东西"
- **不要替学员写任务书。** 可以提问引导，但让他们自己组织语言
- 3 分钟后：放行开始敲命令

**助教的角色：**
- 帮助遇到技术问题的学员（终端操作、文件路径等）
- 观察学员的任务书质量，常见困惑课后反馈给老师

**检查点（1:30）：** "已经看到自己的 Naval 金句学习页面的同学举手！" —— 目标：90%+

**未成功的学员：**
- 助教帮助排查（通常是路径问题或 Claude Code 未启动）
- 确保每人都生成出了页面再进入下一环节

#### 练习 2：修改你的 Naval 学习页面（20 分钟）

**老师说：**

> 现在看看你的页面。有没有什么地方不满意？
>
> 挑的句子太难？音标格式不统一？翻译太生硬？解读太浅？页面不够像学习工具？
>
> 用我们刚学的第四个思维模型 —— **迭代**。告诉 Claude 你想改什么。
>
> 目标：至少改两次。每次只改 1-2 个地方。

**投影上显示修改示例（供参考，不要求照做）：**

| 你觉得... | 你可以这样跟 Claude 说... |
|-----------|-------------------------|
| 句子太难 | "把 5 句话换成更短、更适合英语初学者背诵的句子" |
| 想加内容 | "每句话下面再补一行关键词解释，告诉我这句话里最值得学的 1-2 个词" |
| 布局太挤 | "每张句子卡片之间加大间距，每句的原文、音标、翻译、解读分层显示" |
| 想加效果 | "点击卡片后再展开中文翻译和解读，先默认只显示英文原句" |
| 不够像学习工具 | "这个页面看起来还不像英语学习产品，帮我美化一下，参考简洁的背单词 App 风格" |

**老师在走动时的教学时刻（teachable moments）：**

- 如果学员说"改得不对" → 引导他们描述"我想要的是 X，但你给了 Y" → 这就是"评估"能力
- 如果学员说"它做了我没要求的事" → 引导："你可以说'不要改 X，只改 Y'" → 这就是给 AI 设边界
- 如果学员连续修改效果很好 → 在他们旁边肯定："看到了吗？你已经在用迭代了，这就是核心技能"

**助教此时的角色：**
- 走动观察，看到有趣的成果记下来
- 看到做得好的学员，鼓励他们给邻座看看
- 制造小范围的交流和惊喜

### 第五部分：分享 + 收尾（1:50 - 2:00）

#### 同伴展示（5 分钟）

**老师说：**

> 好，时间差不多了。现在转向你旁边的同学，把你的 Naval 学习页面给 ta 看。
>
> 互相看看，说两句："你这个怎么做的？""你跟 Claude 说了什么？"

**不要做全班展示** —— 第 2 次课还太早，学员的作品差异大，全班展示可能让做得简单的学员尴尬。同伴展示更安全、更轻松。

#### 总结（3 分钟）

**老师说：**

> 今天我们学了四个思维模型：
>
> **第一，你是经理，Claude 是你的初级开发者。** 你给方向、审查结果、提反馈。
>
> **第二，上下文决定一切。** 你说得越清楚，它做得越好。
>
> **第三，先写任务书，再让 AI 动手。** 动手之前花 2-3 分钟想清楚你要什么。
>
> **第四，迭代，不要追求一次完美。** 先做出来，再改好。
>
> 而且你们刚刚做到了 —— 每个人都写了任务书、用 AI 基于 txt 文件做出了自己的 Naval 金句学习页面，还改了至少两次。
>
> 下次课，我们开始做一个更大的东西 —— 一个真正的小工具。我会带着你们一步一步做。

## 关键提醒

1. **对比演示是本课的灵魂。** 差的上下文 vs. 好的上下文，结果差异必须让学员"哇"出来。提前测试至少 3 次，选效果最好的版本。
2. **思维模型要用类比，不要用术语。** "经理管下属"、"改作文"、"装修房子先做大框架再调细节" —— 用学员生活中的事情来解释。
3. **练习时间是神圣的。** 思维模型讲解如果超时，砍讲解而不是砍练习。动手做的记忆是听讲的 10 倍。
4. **不要替学员写提示词。** 提问引导比直接给答案更有价值。"你想要什么风格？" 比 "你应该写'简约风格'" 更好。
5. **第 2 次课是信心关键课。** 第 1 次课的震撼可能伴随着"我能做到吗"的怀疑。第 2 次课的目标是把怀疑转化为"我确实能做到"。确保每个学员都带着一个自己做的页面离开。
6. **同伴展示优于全班展示。** 这个阶段学员作品质量差异大，同伴展示更安全。全班展示留给第 6 次课和第 8 次课。
